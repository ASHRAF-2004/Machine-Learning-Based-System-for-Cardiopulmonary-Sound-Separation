% Abstract (Executive Summary)
% ------------------------------------------------------------

\begin{abstract}
This report presents the software design of an application-based machine learning system for cardiopulmonary sound separation. Cardiopulmonary recordings usually contain overlapping heart and lung sounds together with environmental noise, which complicates auscultation and downstream analytics. The proposed prototype focuses strictly on sound separation (not diagnosis) and provides a reusable Python-based component that receives a mixed chest audio recording and outputs separated heart and lung signals using NeoSSNet. The design is grounded in three recent and strongly related papers: NeoSSNet, which provides a state-of-the-art Conv-TasNet-style separation architecture; a deep learning review for heart sound analysis; and a focused review of heart--lung sound separation methods. Building on these works, the system is structured using a layered architecture and three design patterns---Strategy, Abstract Factory, and Observer---to achieve modularity, maintainability, and clear separation of responsibilities within a NeoSSNet-only implementation.
\end{abstract}

\section{Introduction}
% ============================================================

\subsection{Abstract}
This project designs and documents a machine learning-based system that separates heart and lung sounds from single-channel cardiopulmonary recordings. It is an application-based, software engineering-focused prototype whose core contribution is a clean, pattern-oriented design around a NeoSSNet-based separation pipeline implemented in Python.

\subsection{Problem Statement}
Cardiopulmonary sounds (heart and lung) are generally recorded together using a stethoscope or electronic sensor, and their frequency bands overlap. Classical processing techniques such as bandpass filtering, wavelet-based denoising, adaptive noise cancellation, independent component analysis, and nonnegative matrix factorisation often degrade when signals overlap strongly or when environmental noise is high.\citep{sun2024review}

Recent work shows that deep learning methods can model complex temporal and spectral patterns in heart sounds more effectively than many traditional approaches, but most published systems are presented as research models rather than reusable software applications.\citep{zhao2024deep} There is therefore a gap between advanced cardiopulmonary sound separation research and robust software design suitable for integration into broader HealthTech solutions.

\subsection{Project Objectives}
The project is guided by the following objectives:
\begin{enumerate}[nosep]
  \item Study existing techniques for cardiopulmonary sound separation, with emphasis on NeoSSNet and related findings in recent literature.
  \item Design and develop a Python-based machine learning system that performs cardiopulmonary sound separation using a NeoSSNetPipeline.
  \item Implement and test the prototype using public cardiopulmonary datasets and a clearly documented NeoSSNet configuration.
  \item Evaluate NeoSSNet separation performance using objective metrics (e.g., SDR, SI-SDR) and document the software design using use cases, diagrams, and design patterns.
\end{enumerate}

\subsection{Literature Review}
\textbf{Theme and methodological background.} Cardiopulmonary recordings combine heart and lung sounds with additional noise sources, and the components can occupy overlapping frequency bands. Sun et al. (2024) review heart--lung sound separation research and report that classical methods such as filtering, adaptive noise cancellation, independent component analysis, and NMF-based techniques often struggle under severe overlap and high noise conditions.\citep{sun2024review} Their review documents a transition toward deep-learning-based approaches that can learn stronger representations, while also highlighting real-world generalisation challenges.\citep{sun2024review}

\noindent\textbf{Deep learning in heart sound analysis.} Zhao et al. (2024) provide a broad review of deep learning for heart sound analysis, including convolutional neural networks, recurrent networks, transformers, and hybrid models.\citep{zhao2024deep} They discuss waveform-based and time--frequency approaches across segmentation, classification, denoising, and separation-oriented tasks. The review indicates that deep models often yield stronger performance when sufficient data and preprocessing are available, while practical issues remain in dataset scale, domain shift, and deployment readiness.\citep{zhao2024deep}

\noindent\textbf{NeoSSNet as the implementation foundation.} Poh et al. (2024) introduce NeoSSNet, a real-time neonatal chest sound separation network with an encoder--mask--decoder architecture inspired by Conv-TasNet. The model uses a 1D convolutional encoder, a convolution--transformer mask generator, and a 1D transposed convolution decoder to reconstruct heart and lung waveforms from single-channel chest recordings.\citep{poh2024neossnet} In their evaluation, NeoSSNet outperforms NMF and NMCF baselines in objective separation metrics and computational efficiency, supporting its use as a practical preprocessing component in health monitoring pipelines.\citep{poh2024neossnet}

\noindent\textbf{Identified gap and project positioning.} Although these papers establish the effectiveness of modern separation models, they primarily emphasise algorithmic performance and experimental outcomes.\citep{sun2024review,zhao2024deep,poh2024neossnet} This project addresses the software engineering gap by specifying a NeoSSNet-only system architecture with clear responsibilities, traceable requirements, and maintainable design patterns for a prototype application.

\subsection{Project and System Scope}
The scope of this project is:
\begin{itemize}[nosep]
  \item Build a working \emph{prototype} system in Python that separates mixed cardiopulmonary audio into heart-only and lung-only signals using NeoSSNet.
  \item Focus only on sound separation; no disease diagnosis or clinical decision support is performed.
  \item Use public datasets for training/testing and restrict data volume to what is manageable within the FYP timeframe.
  \item Provide a modest user interface or API for uploading recordings, executing separation, viewing results, and downloading separated outputs.
\end{itemize}

\section{Requirements}
% ============================================================

\subsection{Software Design Concepts and Design Principles}
The system design adopts abstraction, modularity, and separation of concerns. Separation logic is encapsulated in a dedicated pipeline abstraction, while the service layer focuses on workflow orchestration, persistence, and notification.

\subsubsection*{Functional Requirements}
\begin{itemize}[nosep]
  \item The system shall allow authenticated users to upload chest audio recordings.
  \item The system shall execute cardiopulmonary sound separation using \texttt{NeoSSNetPipeline}.
  \item The system shall output separated heart and lung audio signals for each processed recording.
  \item The system shall store recordings, NeoSSNet model configuration, metrics, and separation results for retrieval.
  \item The system shall provide an interface or API to view and download separated outputs.
\end{itemize}

\subsubsection*{Non-Functional Requirements}
\begin{itemize}[nosep]
  \item \textbf{Modularity:} Core components (service, pipeline, repository, notification) should remain loosely coupled.
  \item \textbf{Extensibility:} Internal abstractions should permit future engineering extensions while keeping the current implementation strictly NeoSSNet-based.
  \item \textbf{Performance:} Inference time for a 10-second recording should be within seconds on typical hardware, in line with NeoSSNet reports.\citep{poh2024neossnet}
  \item \textbf{Maintainability:} Code should follow clear naming conventions and layered organisation (UI/API, service, pipeline, persistence).
  \item \textbf{Reproducibility:} Experiments and evaluations should be repeatable through versioned datasets and configuration files.
\end{itemize}

\subsubsection*{Design Principles}
\begin{itemize}[nosep]
  \item \textbf{Single Responsibility Principle (SRP):} Each class (e.g., \texttt{SeparationService}, \texttt{NeoSSNetPipeline}) has a focused role.
  \item \textbf{Open/Closed Principle (OCP):} The service depends on abstraction boundaries, reducing modification risk when internal components evolve.
  \item \textbf{Dependency Inversion Principle (DIP):} High-level modules depend on interfaces rather than concrete utility implementations.
  \item \textbf{Composition over Inheritance:} Services compose pipelines, repositories, and notifiers instead of using deep inheritance hierarchies.
\end{itemize}

\section{Proposed Design Patterns}
% ============================================================

The assignment requires three design patterns. This section presents each pattern with affected components, data flow, UML placeholders, sample code, benefits, and limitations under a NeoSSNet-only system scope.

% ------------------------------------------------------------
\subsection{Design Pattern 1: Strategy}
% ------------------------------------------------------------

\subsubsection*{Function or Software Component Affected}
\begin{itemize}[nosep]
  \item \texttt{SeparationPipeline} interface.
  \item Concrete strategy: \texttt{NeoSSNetPipeline}.
  \item \texttt{SeparationService} as the context that invokes \texttt{SeparationPipeline}.
\end{itemize}

\subsubsection*{Workflow / Data Flow}
\begin{enumerate}[nosep]
  \item The controller receives an upload request and initializes a separation job.
  \item \texttt{SeparationService} obtains a configured \texttt{NeoSSNetPipeline} instance.
  \item The service passes the preprocessed waveform and sampling rate to \texttt{separate()}.
  \item \texttt{NeoSSNetPipeline} executes inference and returns heart and lung waveforms.
  \item The service stores outputs and returns a structured result to the caller.
\end{enumerate}

\subsubsection*{Sample Potential Code}
\begin{lstlisting}[style=pystyle,caption={Strategy abstraction and NeoSSNet implementation.}]
from abc import ABC, abstractmethod
import numpy as np

class SeparationPipeline(ABC):
    @abstractmethod
    def separate(self, waveform: np.ndarray, sr: int) -> tuple[np.ndarray, np.ndarray]:
        """Return (heart_waveform, lung_waveform)."""
        raise NotImplementedError

class NeoSSNetPipeline(SeparationPipeline):
    def __init__(self, model):
        self.model = model  # loaded NeoSSNet model

    def separate(self, waveform: np.ndarray, sr: int):
        heart, lung = self.model.forward_separate(waveform, sr)
        return heart, lung
\end{lstlisting}

\subsubsection*{Benefits}
\begin{itemize}[nosep]
  \item Maintains clean separation between orchestration logic and model inference logic.
  \item Improves testability and maintainability of the NeoSSNet processing component.
\end{itemize}

\subsubsection*{Limitations}
\begin{itemize}[nosep]
  \item Adds an abstraction layer that may appear lightweight for a small prototype.
  \item Requires disciplined interface contracts between service and pipeline layers.
\end{itemize}

\subsection{Design Pattern 2: Abstract Factory}
% ------------------------------------------------------------

\subsubsection*{Function or Software Component Affected}
\begin{itemize}[nosep]
  \item \texttt{PipelineFactory} abstract factory.
  \item Concrete factory: \texttt{NeoSSNetFactory}.
  \item \texttt{ModelConfig} objects used to initialize NeoSSNet components.
\end{itemize}

\subsubsection*{Workflow / Data Flow}
\begin{enumerate}[nosep]
  \item The system reads a \texttt{ModelConfig} record from a configuration file or database.
  \item \texttt{SeparationService} requests a pipeline from \texttt{NeoSSNetFactory}.
  \item \texttt{NeoSSNetFactory} loads checkpoints and runtime settings.
  \item The factory returns a configured \texttt{NeoSSNetPipeline}.
  \item The service executes separation through the returned pipeline instance.
\end{enumerate}

\subsubsection*{Sample Potential Code}
\begin{lstlisting}[style=pystyle,caption={Abstract Factory for NeoSSNet pipeline creation.}]
from abc import ABC, abstractmethod

class ModelConfig:
    def __init__(self, name: str, version: str, params: dict):
        self.name = name
        self.version = version
        self.params = params

class PipelineFactory(ABC):
    @abstractmethod
    def create_pipeline(self, config: ModelConfig) -> SeparationPipeline:
        raise NotImplementedError

class NeoSSNetFactory(PipelineFactory):
    def create_pipeline(self, config: ModelConfig) -> SeparationPipeline:
        model = load_neossnet_model(config.params["checkpoint_path"])
        return NeoSSNetPipeline(model)
\end{lstlisting}

\subsubsection*{Benefits}
\begin{itemize}[nosep]
  \item Encapsulates NeoSSNet component initialization, including checkpoint loading and configuration validation.
  \item Centralizes creation logic, improving consistency across testing and deployment environments.
\end{itemize}

\subsubsection*{Limitations}
\begin{itemize}[nosep]
  \item Introduces additional abstraction that must be justified in a prototype context.
  \item Requires clear documentation of configuration fields to avoid misconfiguration.
\end{itemize}

\subsection{Design Pattern 3: Observer}
% ------------------------------------------------------------

\subsubsection*{Function or Software Component Affected}
\begin{itemize}[nosep]
  \item \texttt{NotificationCenter} as the subject.
  \item Observers such as \texttt{WebSocketClient}, \texttt{EmailNotifier}, and \texttt{AuditLogger}.
  \item \texttt{SeparationService}, which triggers notifications after saving results.
\end{itemize}

\subsubsection*{Workflow / Data Flow}
\begin{enumerate}[nosep]
  \item Observers register with \texttt{NotificationCenter}.
  \item After separation completes, \texttt{SeparationService} calls \texttt{notify\_result\_ready(recordingId, resultId)}.
  \item \texttt{NotificationCenter} iterates over observers and calls update methods.
  \item Observers update UI status, send alerts, or log audit events.
\end{enumerate}

  \centering
  \fbox{\parbox{0.88\textwidth}{\centering
  \textbf{PLACE UML NOTATION HERE (SEQUENCE/STATE DIAGRAM):}\\[0.5em]
  Show SeparationService invoking NotificationCenter, which notifies multiple observers.}}
  \caption{Observer pattern UML placeholder.}

\subsubsection*{Sample Potential Code}
\begin{lstlisting}[style=pystyle,caption={Observer pattern for result notifications.}]
from abc import ABC, abstractmethod

class ResultObserver(ABC):
    @abstractmethod
    def on_result_ready(self, recording_id: str, result_id: str):
        pass

class NotificationCenter:
    def __init__(self):
        self._observers: list[ResultObserver] = []

    def attach(self, observer: ResultObserver):
        self._observers.append(observer)

    def notify_result_ready(self, recording_id: str, result_id: str):
        for obs in self._observers:
            obs.on_result_ready(recording_id, result_id)
\end{lstlisting}

\subsubsection*{Benefits}
\begin{itemize}[nosep]
  \item Decouples separation logic from notification mechanisms.
  \item Allows additional notification channels without changing core processing logic.
\end{itemize}

\subsubsection*{Limitations}
\begin{itemize}[nosep]
  \item Requires robust error handling so observer failures do not break the main workflow.
  \item Excessively heavy observer actions can impact perceived responsiveness.
\end{itemize}
